## Main Conversion Goal

**Get curlers from game plan → game log → season insight as fast as possible.**

The critical conversion funnel is:

```
Plan game day → Log game result → Review season stats → Adjust strategy
```

---

## Three Most Important Workflows

### 1. Game Day (plan → play → log)
```
Open Planner → Fill pre-game → Play game → Fill post-game → View in Game Log
```
**Frequency:** Every game day (1-3x per week in season)
**Current steps:** 8-12 interactions (planner auto-saves)
**Pain:** No direct "Create game from planner" CTA; user must separately log game

### 2. Season Review (stats → insight)
```
Open Dashboard → Check season stats → Filter by range → Inspect position/rink splits
```
**Frequency:** Weekly or after streaks
**Current steps:** 2-3 interactions
**Pain:** Stats grid renders even when empty; filter state lost on view switch

### 3. Ice Memory (log → recall)
```
Add Ice Note after game → Search rink next time → Review historical speed/curl
```
**Frequency:** Every game, consulted before next game at same rink
**Current steps:** 3-4 interactions
**Pain:** Rink memory is view-only; no "typical conditions" summary

## Top Usability Findings (Ranked by Severity)

### S1 — Critical (blocks or degrades core workflow)

| # | Finding | Impact | Location |
|---|---------|--------|----------|
| 1 | **No unsaved-changes guard on modal close** | User fills a modal, accidentally clicks outside or Escape — data lost with no warning | actions.js:174-177 |
| 2 | **Calendar/search filter state lost on view switch** | Returning to Calendar resets filter to "All"; users re-apply same filter every time | bootstrap.js:32 |
| 3 | **No "Create game from planner" CTA** | After planning a game day, user must separately navigate to Games and log it manually; no bridge action | Planner view has no "Log this game" button |
| 4 | **Planner auto-save is completely silent** | Debounced 500ms save gives zero visual confirmation; user doesn't know if data persisted | actions.js:696-698 |

### S2 — High (degrades efficiency or trust)

| # | Finding | Impact | Location |
|---|---------|--------|----------|
| 5 | **Season stats grid renders empty containers when no data** | Four empty insight cards with "No games in this range" text take visual space without value | render.js:11-14 |
| 6 | **Import preview shows counts only, not validation errors** | User sees "5 events, 3 games" but not whether fields are valid or schema version matches | actions.js:458-482 |
| 7 | **Game modal allows partial score entry** | Score can be "7" vs "" (empty); normalization inconsistent between number 0 and empty string | actions.js:255-260 |
| 8 | **No export timestamp display** | User doesn't know when last export was; no "Export data (last: 2 days ago)" indicator | actions.js:393-403 |
| 9 | **Speed dots in Ice modal lack visible focus indicator** | Keyboard users can't tell which speed level is focused | index.html:717-723 |
| 10 | **Game log rows may be hard to expand on mobile** | Expandable detail row has small tap target for expand/collapse toggle | render.js:134-136 |

### S3 — Medium (cosmetic or secondary workflow)

| # | Finding | Impact | Location |
|---|---------|--------|----------|
| 11 | **Checklist has no completion progress indicator** | No bar or percentage showing "4/6 items checked" at a glance | render.js:67-83 |
| 12 | **Modal title doesn't always match action** | "Add Event" persists in title even after validation error and re-open | actions.js:45 |
| 13 | **Danger button color may be borderline WCAG AA** | Granite red (#b44a2c) on light red bg ≈ 3.5:1 at small text sizes | app.css:247-256 |
| 14 | **No breadcrumb or "you are here" beyond tab highlight** | Deep views (event detail, game report) lack navigation context | All views |
| 15 | **Demo data tracked issues may confuse real users** | Issues like "Offline cache not updating" appear to be real bugs, not demo placeholder content | index.html:459-507 |

## Information Architecture Assessment

### Strengths
- 7-view flat nav is appropriate for the feature count — no nesting needed
- ARIA tab roles properly implemented with `aria-selected` and `aria-controls`
- Hash-based routing maintains browser history
- Keyboard shortcuts (1-6) for instant view switching
- Dashboard serves as effective overview hub with drill-through to each view

### Opportunities
- Calendar → Planner → Game Log is the primary flow but lacks explicit linking between steps
- Practice and Ice Notes are isolated from the game-day flow; could surface contextual suggestions
- Season stats could be promoted to a dedicated analytics view as data grows
- UI Issues tracker is a developer tool shipped to users; consider gating behind a setting

---

## State Management Assessment

### Strengths
- Schema version 3 with proper migration from legacy formats
- `normalizeState()` with strict mode for import validation
- `normalizeEvent()`, `normalizeGame()`, `normalizePractice()`, `normalizeIce()`, `normalizeIssue()` — all defensive
- Separate UI prefs key (`cp_ui_prefs_v1`) avoids polluting data schema
- Planner checklist defaults persisted separately
- Demo data fallback on empty load

### Opportunities
- No conflict resolution if same data edited in two tabs
- No `beforeunload` guard on unsaved modal content
- No storage quota check or warning
- No data versioning beyond schema version (no per-record updatedAt)
- Planner entries keyed by date string — no time-of-day differentiation for double-headers

## Visual Design Assessment

### Design System Tokens

| Token | Value | Usage |
|-------|-------|-------|
| `--ice` | #e8f4f8 | Background |
| `--stone` | #2c3e4a | Primary text, buttons |
| `--pebble` | #8ba8b5 | Labels, tertiary |
| `--granite` | #b44a2c | Error, loss badge |
| `--leaf` | #3a7d5c | Success, win badge |
| `--sky` | #1a6fa0 | Primary accent, focus |
| `--white` | #fafcfd | Surfaces |

### Typography
- Headings: Playfair Display (serif)
- Body: DM Sans (sans-serif)
- Data/labels: DM Mono (monospace)

### Component Patterns
- Consistent card system with 22px padding and 16px border-radius
- Three button tiers: primary (stone), ghost (frosted), danger (granite)
- Badge system: result (W/L/D), position, event type, severity
- Pill components for metadata chips
- Stat cards with kicker labels and sparklines
- Toast notifications with optional undo action button
- Transition timing tokens: fast (140ms), medium (220ms), slow (320ms)

### Strengths
- Cohesive curling-themed palette (ice/stone/pebble)
- Consistent spacing (multiples of 4px)
- Smooth animations with overshoot easing
- All touch targets meet 44×44px minimum

## Architecture Notes

### Split-File Structure

| File | Lines | Purpose |
|------|-------|---------|
| index.html | ~841 | All HTML markup, modals, form templates |
| app.css | ~1,688 | All styling, layout, animations, responsive |
| core.js | ~506 | State schema, normalization, storage layer |
| render.js | ~779 | DOM rendering for all 7 views |
| actions.js | ~526 | Modal flows, CRUD handlers, save/delete/export/import |
| bootstrap.js | ~369 | Event wiring, keyboard shortcuts, initialization |
| utils.js | ~371 | Formatting, parsing, UI helpers, animations |

**Load order:** utils → core → render → actions → bootstrap

### Strengths
- Clear separation of concerns (state / rendering / actions / wiring)
- No circular dependencies
- No framework; all vanilla JS
- Easy to locate feature code by file

### Opportunities
- All functions are globals (no module system)
- CSS not split by feature
- No verify-split harness for regression (mentioned but not found)
- core.js could split further into state.js and storage.js
